# OpenAI 联网可用性测试设计文档

## 目标

为 `crates/core/src/api/openai.rs` 增加一个可通过 `cargo test` 运行的真实联网集成测试，用于验证当前配置的 OpenAI 兼容接口是否可用。

该测试的目标不是做完整功能覆盖，而是尽快回答两个问题：

1. 当前 `OPENAI_API_KEY` 与 `OPENAI_BASE_URL` 是否有效
2. 当前模型是否能够成功处理一组最小 `messages` 请求并返回结果

## 范围

本次设计仅覆盖 `OpenAIClient` 的真实联网可用性验证，不扩展到以下内容：

- 不新增 CLI 调试命令
- 不新增额外手动测试脚本
- 不覆盖所有错误分支或重试分支的联网验证
- 不引入复杂快照断言
- 不改造为完整的端到端测试框架

## 推荐方案

采用**显式启用的真实联网集成测试**。

测试默认不强制所有环境执行。当缺少测试所需环境变量时，测试直接跳过；当环境变量齐全时，测试真实请求 OpenAI 兼容接口，并断言返回结果满足最小可用性条件。

这样可以同时满足以下要求：

- 让用户本地快速验证接口能不能用
- 不影响日常 `cargo test` 的稳定性
- 对现有生产代码改动最小
- 保持测试目标聚焦在“联网可用性”而不是“行为全覆盖”

## 方案对比

### 方案 A：显式启用的真实联网集成测试（推荐）

做法：

- 在 OpenAI 相关测试中新增一个 `#[tokio::test]`
- 通过环境变量决定是否执行真实联网测试
- 构造最小 `messages` 请求
- 调用真实 `/v1/chat/completions`
- 验证返回存在文本结果

优点：

- 最符合“试试能不能用”的目标
- 能同时验证 URL、鉴权、模型可达性与响应格式
- 代码改动最少

缺点：

- 依赖外部网络、额度和服务稳定性
- 不适合作为强制 CI 检查

### 方案 B：提炼一个专用测试辅助入口，再由测试调用

做法：

- 增加一个较薄的辅助函数，专门发送固定测试消息
- 联网测试调用该辅助函数

优点：

- 复用性稍好
- 测试代码更短

缺点：

- 引入额外接口，超出当前最小需求
- 容易为一次性验证需求增加不必要抽象

### 方案 C：通过更上层入口间接测试

做法：

- 从 CLI 或服务端更高层入口触发 OpenAI 调用
- 检查最终输出

优点：

- 更接近真实用户路径

缺点：

- 排障成本高
- 不能聚焦验证 `openai.rs` 这一层

## 设计细节

### 测试位置

优先放在 `crates/core/src/api/openai.rs` 的测试模块中，保持与被测实现靠近，便于直接访问现有公开接口。

如果该文件现有测试组织不适合承载联网测试，也可以放入 `crates/core/tests/` 下的集成测试文件，但本次优先选择靠近实现的位置。

### 执行前提

测试依赖以下环境变量：

- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`
- `OPENAI_LIVE_TEST_MODEL`

可选环境变量：

- `OPENAI_LIVE_TEST_SYSTEM`：覆盖默认 system 提示词
- `OPENAI_LIVE_TEST_PROMPT`：覆盖默认 user 提示词

默认值设计：

- system: `你是一个接口联通性测试助手。`
- user: `请只回复 pong`

若缺少必需环境变量，测试直接返回并打印中文跳过说明，不视为失败。

### 测试输入

测试请求使用一组最小 `messages`：

1. system 消息：约束模型只做简单回复
2. user 消息：要求回复简短固定内容

测试目标是验证链路，而不是要求模型严格逐字输出，因此不强制断言返回必须等于 `pong`。

### 断言策略

测试成功条件：

1. 客户端初始化成功
2. HTTP 请求成功完成
3. 返回至少包含一段非空文本内容

测试失败条件：

1. 鉴权失败
2. 地址错误或网络不可达
3. 模型不可用
4. 响应结构异常，未提取到文本内容

断言以“最小可用性”为原则，只验证接口确实能完成一次真实对话请求。

### 错误处理

- 缺少环境变量：跳过测试并输出中文说明
- 请求失败：测试失败，并保留底层错误上下文
- 返回为空：测试失败，并给出中文断言说明

### 与 TDD 的关系

后续实现阶段严格遵循 TDD：

1. 先写一个描述真实联网测试期望行为的失败测试
2. 运行该测试，确认其因测试入口或可测试性不足而失败
3. 仅补最小实现，使测试通过
4. 重新运行测试确认转绿

如果现有 `OpenAIClient` 已足够支撑联网测试，则最小实现可能只涉及测试代码本身，不应附带无关重构。

## 代码结构影响

预期变更范围很小：

- 主要修改：`crates/core/src/api/openai.rs`
- 可能新增：仅当测试组织需要时，新增一个小型测试文件

不修改对外 API 设计，不改变现有 OpenAI 客户端的生产调用路径。

## 验证方式

建议使用单独命令运行该测试，避免噪声：

```bash
cargo test openai -- --nocapture
```

如果需要更精确，也可以运行具体测试名。

执行结果分三类：

- **跳过**：本机未配置联网测试环境变量
- **通过**：真实接口可用
- **失败**：配置、网络、鉴权、模型或响应结构存在问题

## 风险与取舍

### 风险

- 外部服务不稳定会导致测试波动
- 模型输出具有非确定性
- 不同 OpenAI 兼容实现可能在细节上有差异

### 取舍

为降低脆弱性，本设计不做严格文本匹配，只检查最小成功信号，即“请求成功且返回非空文本”。

这样既保留了真实联网验证价值，也尽量避免因模型随机性造成误报。

## 成功标准

满足以下条件即视为设计成功：

1. 用户可通过 `cargo test` 触发该测试
2. 环境变量缺失时不会误报失败
3. 环境变量齐全时会真实请求 OpenAI 兼容接口
4. 用户能从测试结果直接判断“当前接口配置能不能用”
